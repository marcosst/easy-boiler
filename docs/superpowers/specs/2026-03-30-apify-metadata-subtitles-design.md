# Apify Metadata & Subtitles Integration

## Summary

When saving a YouTube video to the library, the system calls the Apify `streamers/youtube-scraper` actor to fetch video metadata and subtitles. Metadata is stored as full JSON in `library_items.metadata`. Subtitles are formatted with `[HH:MM:SS]` timestamps and saved to `midias/{username}/subtitles/{video_id}.txt`, with the path stored in `library_items.subtitle_path`. During this process, a full-screen overlay spinner covers the modal. If the Apify call fails, the save is blocked and the user sees an error with the option to retry.

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| When to call Apify | During save (not preview) | Preview stays fast for URL validation; Apify wait happens after user commits |
| On Apify failure | Block save | Metadata and subtitles are essential for the next phase |
| Metadata scope | Full JSON from Apify | Future needs are not yet defined; store everything |
| Implementation approach | Synchronous call in endpoint | Simplest approach; async `to_thread` prevents event loop blocking; HTMX handles the wait naturally |

## Flow

1. User pastes YouTube URL, clicks "Buscar vídeo" → preview (unchanged)
2. User confirms name, clicks "Salvar na biblioteca"
3. Frontend shows overlay with large spinner over the modal content
4. Backend calls Apify `streamers/youtube-scraper` with the video URL
5. Extracts metadata (full JSON) and subtitles (SRT → `[HH:MM:SS]` format)
6. Saves subtitle file to `midias/{username}/subtitles/{video_id}.txt`
7. Inserts row into `library_items` with `metadata` and `subtitle_path` populated
8. Returns `library_item.html` partial → frontend closes modal
9. On failure: returns error HTML, frontend removes overlay, button re-enables

## Backend

### New function: `fetch_apify_data`

Located in `main.py`. Responsibilities:

- Receives the YouTube video URL
- Builds the Apify actor run payload (same structure as `app/teste-apify.py`):
  - Actor: `streamers/youtube-scraper`
  - `downloadSubtitles: true`, `subtitlesLanguage: "pt"`, `subtitlesFormat: "srt"`
  - `preferAutoGeneratedSubtitles: true`
- Executes the HTTP call via `urllib.request.urlopen` wrapped in `asyncio.to_thread` (non-blocking)
- Reads `APIFY_API_TOKEN` from environment
- Timeout: 180 seconds (from `APIFY_YOUTUBE_TIMEOUT_SECS` env var, default 180)
- Returns `(metadata_dict, subtitle_text)` where:
  - `metadata_dict`: the full item dict from Apify response (first item in the array)
  - `subtitle_text`: formatted subtitle string, or empty string if no subtitles available
- Raises exceptions on failure (network, timeout, invalid response)

### Subtitle formatting

Adapted from `app/teste-apify.py`'s `format_srt_transcript`, but with `[HH:MM:SS]` timestamps:

```
[00:00:05] Olá, bem-vindos ao canal
[00:00:12] Hoje vamos falar sobre React Hooks
[00:01:30] Primeiro, vamos entender o useState
```

The SRT timestamp `HH:MM:SS,mmm` is truncated to `HH:MM:SS`. Duplicate consecutive lines are skipped.

### Changes to `POST /htmx/library/save`

For `type == 'youtube'` only (PDF flow unchanged):

1. Extract `video_id` from URL
2. Call `fetch_apify_data(url)` — wrapped in try/except
3. On success:
   - Create directory `midias/{username}/subtitles/` if needed
   - Write subtitle text to `midias/{username}/subtitles/{video_id}.txt`
   - INSERT into `library_items` with `metadata = json.dumps(metadata_dict)` and `subtitle_path = "{username}/subtitles/{video_id}.txt"`
4. On failure:
   - Return the preview form HTML with an error banner at the top
   - HTTP 422 so HTMX routes the response to `hx-target-error` (the preview area) instead of the success target (the items list)

### Error messages

| Scenario | Message |
|----------|---------|
| Token not set | "Token da Apify não configurado." |
| Network/timeout | "Falha ao baixar metadados. Tente novamente." |
| No subtitles | "Nenhuma legenda disponível para este vídeo." |
| Invalid response | "A Apify retornou uma resposta inválida." |

## Frontend

### Overlay spinner in `library_preview.html`

The save form gets an overlay `div` controlled by Alpine.js state:

- **Trigger**: `@htmx:before-request` on the form sets `saving = true`
- **Dismiss**: `@htmx:after-request` sets `saving = false`
- **Overlay**: absolute-positioned div covering the entire form area
  - Background: `bg-black/60` (semi-transparent dark)
  - Spinner: 56px, teal border animation
  - Text: "Baixando metadados e legendas..."
- **Button**: disabled while `saving` is true, text changes to "Salvando..."

On error, the endpoint returns HTTP 422 (or another non-2xx status). The form uses `hx-target-error="#library-preview-area"` so that error responses replace the preview area with the same form plus an error banner, while successful responses still target `#library-items-list` with `beforeend`. The overlay is removed because the HTML is fully replaced.

### No changes to

- `library_add_modal.html` (modal container)
- `library_item.html` (item display)
- `topics.html` (page container)
- Preview endpoint (`POST /htmx/library/preview`)

## File structure

```
midias/
  {username}/
    subtitles/
      {video_id}.txt    ← new subtitle files
```

## Database

No schema changes needed. The `metadata` (TEXT) and `subtitle_path` (TEXT) columns already exist in `library_items` but are currently unused.

## Configuration

All env vars already exist in `.env.example`:

| Var | Default | Purpose |
|-----|---------|---------|
| `APIFY_API_TOKEN` | (required) | Apify authentication |
| `APIFY_YOUTUBE_SCRAPER_ACTOR_ID` | `streamers/youtube-scraper` | Actor to run |
| `APIFY_YOUTUBE_SUBTITLES_LANGUAGE` | `pt` | Subtitle language |
| `APIFY_YOUTUBE_PREFER_AUTO_GENERATED_SUBTITLES` | `true` | Prefer auto-generated |
| `APIFY_YOUTUBE_TIMEOUT_SECS` | `180` | Request timeout |
